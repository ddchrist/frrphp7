Exported Fields Description

Id

Unique number identifying the change request. CR reference.

External_id

An external reference – (e.g. Customer or subcontractor reference).

Program_product

Project origin of the CR (When Project) or

Domain, which will be affected in resolving this problem (When Component).


Refer to APPENDIX 1 for examples of Context Elements.

Element_Category

For a Project CR, field set to NONE.

For a Component CR, Component name according to examples in APPENDIX APPENDIX 1.1

Element_name

For a Project CR, field set to NONE.

For a Component CR, project name of the project that raises the CR, or NONE

State

Current CR status

States are:

  • SUBMITTED

  • ANALYSED

  • ACCEPTED

  • STARTED-WORK

  • SOLVED

  • VERIFIED

  • CLOSED


Headline

Title of the CR

Report_Type

Type of the Change Request

  • Problem Report, when the product does not operate according to the system or allocated (PCR)

  • Enhancement Request, A new feature, which has yet to be developed (ECR)

  • Information Request, questions from a customer for example

  • Development Request, A development task not a PCR, ECR nor Information Request.(e;g; to archive test data) (i.e. Depending of UCM development model, if an ECR is developed by two resources, then two children DCRs could be created from one father ECR)

ATM_Severity

Severity of the defect: from S1 to S5


The severity provides objective criterion to the CR correction priority setting.

Refer to APPENDIX 2 for detailed definitions.

Detected_Release

Version of the software/hardware/document on which the defect occurs.

(Tag: DAT_B2_9, SRS_RevB)

Originator_FullNm

The originator of the CR. (by default Name of person creating CR)

Must be entered if different from that in “PCR_Creator fullnm”

Detection_Origin

Site where CR was detected.

Element_Nature

Type of the objects involved in the Change Request

  • Documentation, when an approved document is impacted.

  • Item, when a software or hardware item is impacted.

Data, when an element other than hardware, software or documents is impacted.

Reproducibility

Type of defect.

  • Systematic, = problem consistently reproducible

  • Random, = problem difficult to reproduce

  • Seen Once = not reproducible, seen once

For Enhancement, Information or Development request, field to set to Systematic.

ATM_Safety

Safety impact of the defect

  • Blank,

  • Not_Safety_Impact,

  • Safety_Impact


Blank is the default value

One of the two last values could only be set and updated by Safety managers or by delegation Safety Peers.

Description

Field used to describe the problem in detail. The originator could also describe the wanted modification. A template could be defined in order to help originator to fill this field.

ATM_Component_List

Component(s) affected by the request.

The component list is established during Analysis phase.

ATM_Document_List

Documentation at Project or Component level affected by the resolution of this request.


ATM_Functionality_List

Keyword used to determine the impacted functionality at project or component level.

This Field can be used to perform a search.


ATM_Impacted_Requirement

Identifies the requirement against which the CR is raised.

It is the responsibility of the V&V Engineer and/or the customer to initially enter the requirement. If no requirement has been entered the Developer shall enter the affected requirement or “Other” where it is not possible to identify a requirement. Even if a System Requirement has been entered it shall not prevent the Developer from entering a related component Requirement.

When the CR has a documentation impact, this field shall contain the list of the requirements updated by this CR (new or modified or removed requirements).

ATM_Phase_Stage

Identifies where the defect was raised (which IBB, which site…)

This field is used in coordination with the Detection Phase and is customisable for each project.

Decision_Meeting_Type

Field used by the Decision Authority Responsible to specify if CR has to be submitted to:

  • JCCB (Joint Configuration Control Board) if customer approbation is needed

  • PCCB (Product Configuration Control Board) if supposed to be a product CR

  • SCCB (Centre Segment Configuration Control Board) if proposed to be discussed at SCCB level

Decision_Date

Date of CR decision at CCB.

Decision_Auth_Rep_FullNm

The CCB Chairman who takes the decision.

Internal_priority

The priority for correction. Could different of the initial priority.

  • Critical: very urgent, need a patch.

  • High: next release.

  • Low: only when all other priorities have been fixed.

  • More Info required: need to get more information from the PCR originator.

  • Postponed: to be reviewed at a later date (Postponed Until date to be filled).

Target_Release

Expected release of the element in to which this change is supposed to be fixed. The Engineering Project Manager or CCB manager will assign a project version.

Postponed_Until

If Postponed set in Internal Priority, Date when the decision meeting will treat again the CR. Deadline to be set.

Decision_Description

Field used by the Decision Authority Responsible to describe the decision in detail. A template could be define in order to help to fill this field

Realisation_Date

When the realisation is available for verification.

Set to the current date automatically.

Realisation_Resp_FullNm

The person who will fix the defect.

Realisation_Release

The version of the delivery in to which this change is going. The Engineering Project Manager will assign a version and will notify the developers of the version number to be used when delivering the files. The system configuration manager to automatically update this field will use this version number.

V/C contains “x.x” when the CR is delivered but not yet integrated in a system release.

Verification_Date

Date of CR realisation has been verified.

Set to the current date automatically.

Verification_Resp_FullNm

Name of the person in charge of the verification.

Verified_Release

The version of the software delivery into which this change is verified.

Verification_OK

Result of the verification (YES or NO).

ATM_Test_Reference

Identifies system test reference.

Type None if not.

If the full test description is in the Description section, type, “Refer to Description”.

Closure_Date

Date of last update of report.

Set to the current date automatically.

Closure_Auth_Rep_FullNm

The person(s) who will close this CR. This will typically be assigned by the validation team.

Delivered_Release

The version of the delivery in to which this change is going.

At component level, will be automatically filled from VDD.

At project level, release in which final testing has been performed.

Closure_Reason

There is one final state: Closed. Closure reason field is used at the decision or closure meeting to explain why the change request is closed. For capitalisation.


  • Realisation, when the change request has been fixed in a new release.

  • Duplication, when the change request is strictly identical to another change request. The processed original change request has to be modified to add the current change request in its “Identical Pcrs” list. Then, the current change request is closed

  • Rejection, when there is no problem. The behaviour is as specified.

  • Information Provided when the change request is finally a question.

  • Pcr Reworked when a cloned change request has been created (life cycle with no back return).

  • Problem Disappeared when the default is no more reproducible, even by the Originator.

  • Postponed when the change request needs to not appear anymore during decision meeting (this is an alternative to put Postponed in Internal Priority and leave the change request Analysed).

Closure_Description

Field used to describe the closure procedure. A template is defined in order to help to fill this field.

Originator_Acceptance

Result of the closure (YES or NO).

ATM_Free_Keywords

Free Field.

Creation_Date

The date of the CR creation.

Field set to the current date automatically.

Pcr_Creator_FullNm

CQ User Name, as specified in configuration file.

Field automatically filled in at CR creation, based on the CQ login ID of the person entering the CR.

Expertise

Filed used to give the profession implied in the change request.

  • System when the change request impacts several components (e.g Interface Requirement Specification impacts, modification of system level documents: SSS, SSDD, IRS, ICD…etc) System change request is created by a System team (integration or validation team) and needs to be validated at system level once all its children component change requests are at least at verified state.

  • Software when the change request impacts a Software component.

  • Hardware when the change request impacts a component of System Infrastructure.

  • Other when the change request impacts another profession, for example information management.

Nota: When Hardware is selected, no software component could be impacted. In that case, the impacted components field has to remain empty.

Hierarchical Links: Children_Pcrs.id

List of the change requests id down in the hierarchy.

This field is automatically filled by the tool.

Hierarchical Links: Parent_Pcrs.id

Identify the CR id which this current CR comes from

This field is automatically filled in by CQ when the PCR is copied.

List of the change requests id describing the same problem.

Contain the links of the linked CRs (up and down link)

List of the change requests id clones of the original CR.

This field is automatically filled by the tool.

Clone Link: Previous_Pcrs.id

Identify the CR id of the CR Cloned.

This field is automatically filled in by CQ when the PCR is cloned.


APPENDIX 1. « context element » examples


The contents of the following three fields

    • Project/Product,

    • Element_Category

    • Element_Name


Determines the CONTEXT element.


The CONTEXT is assigned to the CR at its creation (Submitted State) and is no more modifiable.

Ex 1: At project level:

    • Project/Product, PROJECT_NAME

    • Element_Category None

    • Element_Name None


Ex 2: At Component level, for an Allocated_CR

    • Project/Product, SURV

    • Element_Category SNMAP

    • Element_Name PROJECT_NAME


Ex 3: At Component level, for an internal product CR

    • Project/Product, SURV

    • Element_Category SNMAP

    • Element_Name None



The exact list of DOMAINS and COMPONENTS will be automatically generated, using a specific functionality of CQ tool, and stored as an appendix


APPENDIX 2.« Severity » Definitions

Value

Definition (for guidance)

Examples of Symptoms (for guidance)

S1

Prevents accomplishment of essential capability

  • System unusable (network, time distribution, several essential functions (FDP, SNMAP, Trackers, CDP) unavailable),

  • Unpredictable and frequent ODS crash / freeze with no workaround,

  • Total system crash,

  • Major display error on Track (position, erroneous coupling, level),

  • Frequent simulation exercise restart with no workaround

S2

Adversely affects the accomplishment of an operational capability

  • Total failure to group/de-group positions,

  • System performance below usable threshold on usual Controller action under normal traffic condition,

  • Major error in external data exchange,

  • Essential function unavailable (FDP, SNMAP, Tracker, CDP) longer than a restart time,

  • Major failure in FPL interaction blocking the progress of the control of the flight (assume, inter-sector transfer),

  • Missed alarm in STCA, MSAW

S3

Adversely affects the accomplishment of an operational capability. A workaround solution is available

  • Major error in actions not relative to FP and Track (Areas activation, AIS, …)

  • Major error in FPL interaction, or in FDP processing, not blocking the progress of the control of the flight(s)

  • Major error in display of non Flight Plan nor track info

  • Unable to control the system (node and devices control)

  • Major display error on flight plan / track info other than above

  • Major malfunction of ODS local function

  • Missed alarm in MTCD, RAM, CLAM, APW, SAP

  • Any crash occurring at low frequency, or predictable

S4

Provokes an operator inconvenience or annoyance and does not affect mission essential capability

  1. Minor error in external data exchange

  2. Rare, or minor Flight Plan error not affecting the progress of the control

  3. Minor error in display of non Flight Plan nor track info

  4. Malfunction on non operational functions

  5. Error in monitoring information (node, device)

  6. Any system performance problem not listed above

  7. Any minor display error

  8. Minor malfunction of ODS local function

  9. False alarm in STCA, MSAW, MTCD, RAM, CLAM, APW, SAP

  10. Any configuration / system production and installation error

  11. Any error or malfunction occurring at very low frequency

S5

All other errors

Any problem deemed less significant than the ones described above
